# Implementation Plans — Full-Repo Bug Hunt (Pokenic)

Generated by the `improve` skill on 2026-07-02, against commit `4ca2593`.
Scope: every function / feature / button / route across the **storefront**
(`src/`), the **Medusa backend** (`backend/packages/api`), and the **admin
dashboard** (`backend/apps/admin`). Method: seven parallel read-only audits
(money paths, gameplay/fulfillment, admin API + platform, cross-cutting
security, storefront account/money UI, storefront gameplay/marketing UI, admin
dashboard app), then every surviving finding re-opened and vetted against the
live code by the advisor.

**Payment gateway is deliberately excluded** — top-up runs through the mock
gateway (`ALLOW_MOCK_TOPUP` / `unsafe-demo` prod opt-in) and reward payout is
dormant until a real PSP + AMLA compliance land. Those are decided tradeoffs,
not bugs. Implementation bugs _inside_ those guards were still in scope.

Each executor: read the plan fully before starting, honor its STOP conditions,
and update your row when done. **Do not fix anything not covered by a plan
without checking the "considered and rejected" list first** — several
plausible-looking issues were investigated and found to be by-design.

---

## Headline

The money core is in good shape. Per-customer advisory locks, an append-only
ledger with a DB-level unique index on `pull_id`, idempotent workflow steps,
frozen-account gates, and commission decay all held up under scrutiny. **No
confirmed critical vulnerability and no confirmed money-loss bug were found.**
What remains is one high-value _verification_ (admin FX route auth), one
real UX-severity bug (a charged user can get stuck on the spin screen), and a
tail of low-severity validation/hardening items.

## Findings considered and rejected

Recorded so nobody re-audits them. Each was opened and confirmed by the advisor.

- **Weighted-pick last-row bias** (`modules/packs/pick.ts`): the `roll >= totalWeight` fallback to the last row is standard weighted-selection and documented as an accepted float-drift edge; bias is asymptotically zero. Not a bug.
- **Stock decrement race / negative stock** (`workflows/steps/decrement-card-stock.ts`): explicitly by-design — "STOCK IS A COUNTER, NOT A GATE." The adjust is guarded by `target.stocked > 0`, and the one-unit drift is documented as acceptable. Not a bug.
- **Notification mark-read TOCTOU / duplicate rows** (`store/notifications/[id]/read`): a partial unique index on `(notification_id, customer_id) WHERE deleted_at IS NULL` already exists (`models/notification-read.ts`); the retry-on-conflict re-fetch is correct. Only a rare double-miss re-throw remains (cosmetic).
- **CSRF on state-changing store POSTs** (open, buyback, topup, draw, delivery, referral): customer auth is **bearer-only** — the backend issues no customer session cookie, so there is no ambient-credential CSRF vector. Documented in `middlewares.ts:32-44`. Not a bug.
- **Admin double-submit on money buttons**: `@medusajs/ui` `Button` with `isLoading` renders as `disabled`, and the handlers (`requestAdjust`/`applyAdjust`) early-return while `isPending`. Guarded.
- **Slot `settled` ref "never cleared"** (`SlotReelColumn.tsx`): the main effect sets `settled.current = false` on its first line, and the column remounts per spin (keyed by `spinKey`). Reset is correct.
- **Slot `onSettled` stale closure** (`SlotReelStack.tsx`): `onSettledRef` is refreshed every render via an effect; the callback always reads the current value. Correct pattern.
- **`availableBalance` staleness / SHA-256 idempotency collision / soft-delete replay**: all defended by the per-customer advisory lock in `mutateCreditAtomic` and existing idempotency tests. Not bugs.
- **Media upload path traversal** (`admin/media/route.ts`): `path.basename()` after `\`→`/` normalization is sufficient; multer memory storage + `validateImage` (mime/magic-byte/size/dims) gate the rest.
- **"Verify topup/draw rate limit + CSRF exists"** (raised by the money-path audit): confirmed present in `middlewares.ts` — `createCreditTopupRateLimit()` on `/store/credits/topup`, `deliveryWriteRateLimit` on `/store/rewards/*`, all after `authenticate('customer', ['bearer'])`.

## Direction (not bugs — for the maintainer, low volume by request)

The user asked for bugs, so this is deliberately short:

- **Pagination is capped, not paginated** on several read surfaces (recent-pulls
  hard-capped at 12, vault at 500, no `skip`/`limit` params). Fine today; will
  bite the first user with a large vault or an "infinite scroll" feed. Covered
  as a note in plan 008, not built out.
- **A backend↔storefront enum contract check** (plan 005) is the first instance
  of a general class: several shared Zod schemas mirror backend enums by hand.
  A tiny codegen or shared-constants module would kill this whole drift class —
  worth considering if it recurs.
